Overview

Single PostgreSQL table for storing application configurations across multiple environments (dev, test, uat, prod). The config_name column allows each application to have multiple named config groups — keeping concerns separated without multiplying tables. Azure Key Vault URIs are stored directly inside the config JSONB and resolved at runtime by the application or Ansible.

Design Principles

PrincipleDetail
🗂️ Single tableNo joins, minimal ops overhead, easy to reason about
🏷️ config_name groupingOne row per logical concern — database, cache, feature-flags, integrations
📦 JSONB config columnFlexible, schema-free — no DB migrations when config structures change
🔒 Vault URIs in-placeKey Vault URIs live alongside regular values in config. Resolved at startup by app / Ansible
📝 Version counterIncremented on every update — supports audit trail and optimistic locking
☁️ Azure-nativeManaged Identity auth for Key Vault — no credentials stored anywhere

Schema Definition

sql — configurations table 
CREATE TABLE configurations (
    id              UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    app_name        VARCHAR(150) NOT NULL,
    config_name     VARCHAR(150) NOT NULL,   -- 'database', 'cache', 'feature-flags', 'integrations'
    environment     VARCHAR(50)  NOT NULL,   -- 'dev', 'test', 'uat', 'prod'
    config          JSONB        NOT NULL DEFAULT '{}',
                                               -- Key Vault URIs stored inline alongside regular values

    version         INTEGER      NOT NULL DEFAULT 1,
    is_active       BOOLEAN      NOT NULL DEFAULT TRUE,
    created_by      VARCHAR(150) NOT NULL,
    updated_by      VARCHAR(150),
    created_at      TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ  NOT NULL DEFAULT NOW(),

    UNIQUE (app_name, config_name, environment)
);

-- Indexes
CREATE INDEX idx_conf_app_env  ON configurations (app_name, environment);
CREATE INDEX idx_conf_app_name ON configurations (app_name, config_name, environment);
CREATE INDEX idx_conf_gin      ON configurations USING GIN (config);

Common Query Patterns

List all config names for an app

sql 
SELECT config_name, environment, version, updated_at
FROM   configurations
WHERE  app_name  = 'trading-engine'
  AND  is_active = TRUE
ORDER BY environment, config_name;

Fetch a specific config at runtime

sql 
SELECT config, version
FROM   configurations
WHERE  app_name    = 'trading-engine'
  AND  config_name = 'database'
  AND  environment = 'prod'
  AND  is_active   = TRUE;

Update a single key (non-destructive)

sql 
UPDATE configurations
SET    config     = jsonb_set(config, '{max_connections}', '200'),
       updated_by = 'pradeep',
       updated_at = NOW(),
       version    = version + 1
WHERE  app_name    = 'trading-engine'
  AND  config_name = 'database'
  AND  environment = 'prod';

Rotate a secret — point to new Key Vault version

sql 
UPDATE configurations
SET    config     = jsonb_set(
                       config,
                       '{password}',
                       '"https://myvault.vault.azure.net/secrets/trading-db-password/newversion123"'
                   ),
       updated_by = 'pradeep',
       updated_at = NOW(),
       version    = version + 1
WHERE  app_name    = 'trading-engine'
  AND  config_name = 'database'
  AND  environment = 'prod';

Find all configs containing a Key Vault URI (audit)

sql 
SELECT app_name, config_name, environment
FROM   configurations
WHERE  config::text LIKE '%vault.azure.net%'
  AND  is_active = TRUE;

Secret Handling

⚠️
Never store plaintext secrets in the config column. Use Azure Key Vault URIs as the value. The application and Ansible automatically detect and resolve any string containing vault.azure.net at startup.

✅ Correct — Key Vault URI as value

{
  "host": "db-prod.example.com",
  "port": 5432,
  "password": "https://myvault
    .vault.azure.net
    /secrets/db-password"
}

❌ Wrong — plaintext secret in config

{
  "host": "db-prod.example.com",
  "port": 5432,
  "password": "Sup3rS3cret!@#"
}

How resolution works

StepFastAPI (startup)Ansible / AAP
1Load config JSONB from DBQuery configurations table
2Recursively scan all string values for vault.azure.netInline Python script flattens URIs
3Fetch secrets in parallel via Azure SDK (Managed Identity)az keyvault secret show --id <uri>
4Inject plaintext values back at same JSON pathsInject into config dict, template to .env / YAML
5Store in app.state.config — never loggedWritten to file with mode: 0640, no_log: true
🔐
Auth — use Managed Identity (recommended) Assign the VM, VMSS, or AAP instance a Managed Identity with Key Vault Secrets User role. DefaultAzureCredential picks this up automatically — no AZURE_CLIENT_SECRET ever stored anywhere. For local dev, run az login and it works without any extra config.

FastAPI Integration

Set APP_NAME, CONFIG_NAME, and ENVIRONMENT as env vars. The app loads, resolves, and caches config before accepting any traffic. Call POST /config/reload after a Key Vault secret rotation — no restart needed.

python — startup sequence (main.py) 
# ENV VARS: APP_NAME, CONFIG_NAME, ENVIRONMENT, DATABASE_URL

@asynccontextmanager
async def lifespan(app: FastAPI):
    app.state.db = await asyncpg.create_pool(DATABASE_URL)

    row = await _fetch_config(app.state.db, APP_NAME, CONFIG_NAME, ENVIRONMENT)
    # SELECT config, version FROM configurations
    #   WHERE app_name=$1 AND config_name=$2 AND environment=$3 AND is_active=TRUE

    raw_config        = dict(row["config"])
    app.state.config  = await resolve_secrets(raw_config)
    # resolve_secrets() scans all string values for vault.azure.net
    # fetches in parallel, injects plaintext back at same JSON paths

    yield   # app is ready

@app.post("/config/reload")
async def reload_config():
    """Hot-reload after Key Vault secret rotation."""
    row               = await _fetch_config(...)
    app.state.config  = await resolve_secrets(dict(row["config"]))
    return {"status": "reloaded", "version": row["version"]}

Ansible / AAP Integration

The playbook fetch_and_apply_config_secrets.yml queries the DB, resolves Key Vault URIs using az keyvault secret show, and templates the resolved config to .env or YAML files on the target host.

bash — run the playbook 
ansible-playbook fetch_and_apply_config_secrets.yml \
  -e "app_name=trading-engine config_name=database environment=prod"
AAP Auth MethodHow to configure
Managed Identity ✅ recommended Assign the AAP VMSS a Managed Identity with Key Vault Secrets User role — zero config needed in the playbook
Service Principal Create a Custom Credential Type in AAP that injects AZURE_CLIENT_ID, AZURE_TENANT_ID, AZURE_CLIENT_SECRET as env vars — values stay in AAP vault, never in playbook vars
Local dev Run az login — DefaultAzureCredential picks it up automatically
⚠️
AAP activity stream warning Never include resolved secret values in set_stats — they appear in the AAP activity stream in plaintext. Only pass non-secret values (host names, feature flags, versions) downstream to other job templates.

Design Decisions & Trade-offs

DecisionBenefitTrade-off
Single tableZero joins, simple queries, easy opsNo enforced per-key schema
config_name columnSeparates concerns, independent versioning per groupApp needs to know which config_name to load
Vault URIs inside config JSONBNo separate column needed, one place to lookApp/Ansible must always scan for URIs before use
version counterOptimistic locking, audit correlationApp must increment on every update
GIN index on configFast JSONB key/value lookupsSlightly larger index size
Managed Identity authNo credentials stored anywhereRequires Azure-hosted deployment or AAP on Azure
